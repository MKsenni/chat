# Architecture

## 1. Стейт-менеджер: Zustand

**Почему не альтернативы:**

- **Redux / Redux Toolkit** - избыточен для одного экрана. Для масштабного приложения - да, но здесь это overhead.
- **React Context + useReducer** - подходит для простых случаев, но при частых обновлениях (статусы сообщений) вызывает ре-рендеры. Нет встроенной персистентности.

**Почему Zustand:**

- Один стор без провайдеров - подписка через хук `useChatStore`
- Встроенный `persist` middleware для localStorage - персистентность
- Селекторы из коробки предотвращают лишние ре-рендеры

## 2. Структура и слои

```
src/
├── components/          UI-слой (React-компоненты)
│   ├── Chat.tsx            корневой layout чата
│   ├── MessageList.tsx     скроллируемый список сообщений
│   ├── MessageBubble.tsx   пузырь сообщения
│   ├── MessageStatus.tsx   иконки статусов
│   ├── ChatInput.tsx       поле ввода + отправка
│   ├── Avatar.tsx          аватар/инициалы
│   └── TypingIndicator.tsx индикатор «печатает…»
├── assets/icons/        SVG-иконки (через vite-plugin-svgr)
├── store/               Стейт-слой
│   └── chatStore.ts        Zustand store + persist + бизнес-логика
├── services/            Сервисный слой (эмуляция API)
│   └── messageService.ts   sendMessage, getBotReply
├── types/               Типы и контракты
│   └── message.ts          Message, MessageStatus, MessageAuthor
└── utils/               Утилиты
    └── formatTime.ts       форматирование времени
```

**Разделение ответственности:**

- **components/** - только отображение и пользовательские события. Не знают про API.
- **store/** - управление состоянием.
- **services/** - изолированная логика. Легко заменить эмуляцию на реальный API без изменения стора.
- **types/** - общие контракты, от которых зависят все слои.

## 3. Что изменилось бы в production с командой из 5 человек

- **Реальный API** - `services/` переключились бы на fetch/axios с обработкой авторизации, retry-логикой и WebSocket для realtime.
- **Стейт** - вынесение бизнес-логики из стора в отдельные use-case функции или хуки. Zustand остаётся хранилищем.
- **Тестирование** - unit-тесты для стора и сервисов (Vitest).
- **Пагинация** - бесконечный скролл или подгрузка истории. Виртуализация списка через `react-window` при > 500 сообщениях.
- **Обработка ошибок** - Error Boundary на уровне приложения, toast-нотификации, лоигрование.
- **CI/CD** — линтинг, тесты, сборка и деплой в пайплайне. Storybook для компонентов.
- **Монорепо** — разделение на пакеты (ui-kit, api-client, shared-types), если проект растёт.
